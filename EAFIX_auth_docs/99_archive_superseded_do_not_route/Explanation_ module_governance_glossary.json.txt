What 
module_governance_glossary.json
should include
It should define governance vocabulary, not module details. Its job is to make terms deterministic for AI agents before they create MCRs, catalogs, crosswalks, manifests, or validators.
The attached artifact list shows the glossary would sit above artifacts like module_identity_policy.json, module_change_request.schema.json, lifecycle enums, crosswalks, and validators.  
Required top-level sections
{
  "schema_version": "1.0.0",
  "document_type": "module_governance_glossary",
  "status": "canonical",
  "governance_scope": {},
  "authority_rules": {},
  "terms": [],
  "relationships": [],
  "forbidden_interpretations": [],
  "ai_usage_rules": [],
  "versioning_policy": {}
}
Required fields per term
Each glossary term should use this structure:
{
  "term": "canonical_module",
  "display_name": "Canonical Module",
  "category": "module_identity",
  "definition": "An approved module identity locked in the current catalog for the active pass.",
  "applies_to_artifacts": [
    "atomic_module_catalog.current.json",
    "module_system_index.json",
    "atomic_module_crosswalk.current.json",
    "atomic_module_manifests/"
  ],
  "must_not_mean": [
    "implemented module",
    "fully documented module",
    "manifest-ready module"
  ],
  "related_terms": [
    "module_id",
    "canonical_symbol",
    "lifecycle_status",
    "manifest_ready"
  ],
  "governance_rule": "A canonical module may still lack file bindings, runtime bindings, or complete manifests.",
  "example": {
    "correct": "R1_RISK_EVALUATOR is canonical but may still require downstream file or manifest reconciliation.",
    "incorrect": "Canonical means implementation is complete."
  }
}

Creation method
Create it in this order:
	1.	Define metadata
	•	schema version
	•	document type
	•	status
	•	created date
	•	authority scope
	2.	Define authority rules
	•	glossary defines terms only
	•	does not override module catalog
	•	does not override process catalog
	•	does not override MT4 authority
	•	does not override UI catalog
	3.	Add core terms first
	•	module
	•	atomic module
	•	module ID
	•	canonical symbol
	•	lifecycle status
	•	catalog pass
	•	MCR
	•	crosswalk
	•	manifest
	4.	Add forbidden meanings Example:
{
  "term": "canonical",
  "forbidden_meaning": "implemented",
  "reason": "A module can be canonical before source files, runtime bindings, or manifests are complete."
}
	1.	Add relationship rules Example:
{
  "relationship": "module_id_to_canonical_symbol",
  "rule": "module_id is permanent; canonical_symbol is human-readable and may change through an accepted MCR."
}
	1.	Add AI usage rules Example:
{
  "rule_id": "GLOSSARY-AI-001",
  "rule": "When generating governance artifacts, use glossary definitions exactly and do not invent alternate meanings."
}
