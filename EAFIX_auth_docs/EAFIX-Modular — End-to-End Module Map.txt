 EAFIX-Modular — End-to-End Module Map

  ▎ Structured per Module-Centric Repository Best Practices: each module
  ▎  is an atomic context unit with co-located artifacts, a single
  ▎ manifest, and a numeric identity (^5\d{19}$). 28 canonical modules +
  ▎  SHARED_LIBS. The EA-side is a separate deployable (MQL4) decomposed
  ▎  into Systems A–G.

  Group 6 — MT4 Bridge Layer (B-prefix, INTEGRATION_BRIDGE_MODULE)

  Implements the 14-phase Atomic Process Flow v3.0 (CSV-first, socket
  fallback).

  Module ID: B1_MT4_ADAPTER_TRANSPORT
  Symbol: Bridge Transport
  Sub-modules / Key Functions: CSV atomic write (file_seq +
    checksum_sha256 + temp→rename), Socket bridge (port 5555)
  Input → Output: OrderIntent → CSV/socket signal files
  ────────────────────────────────────────
  Module ID: B2_MT4_EA_EXECUTOR
  Symbol: MT4 EA (MQL4 v7.10)
  Sub-modules / Key Functions: EA-side: reads signal CSVs, executes via
    OrderSend(), writes feedback; DLL socket bridge
  Input → Output: CSV/socket → MT4 execution → feedback CSVs
  ────────────────────────────────────────
  Module ID: B3_EXEC_EVENT_NORMALIZER
  Symbol: Exec Event Normalizer
  Sub-modules / Key Functions: Parse MT4 feedback CSVs; normalize into
    ExecutionReport
  Input → Output: Feedback CSV → ExecutionReport → Redis
    reentry_decisions

  Bridge Protocol — 14 Phases:

  ┌───────┬────────────────────────────────────┬────────────────────┐
  │ Phase │                Name                │     Direction      │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 0     │ Bootstrap                          │ Python init        │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 1     │ Config resolve                     │ Python → EA        │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 2–4   │ Signal build + validate + export   │ Python → CSV       │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 5     │ EA poll + receive                  │ EA reads CSV       │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 6–8   │ Reentry request + decision +       │ EA ↔ Python socket │
  │       │ response                           │                    │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 9–11  │ Execution + ACK + result write     │ EA → feedback CSV  │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 12–13 │ Feedback read + chain advance      │ Python reads CSV   │
  ├───────┼────────────────────────────────────┼────────────────────┤
  │ 14    │ Test cues                          │ Python test        │
  │       │                                    │ harness            │
  └───────┴────────────────────────────────────┴────────────────────┘

  Group 7 — Reentry Decision Engine (E-prefix, REENTRY_MODULE)

  Microservice home: reentry-matrix-svc + reentry-engine (port 8085)

  The 4D Reentry Matrix v3.0: 8 signal types × 6 outcomes × 4 proximity
  zones × conditional duration = 652 combinations, max chain O→R1→R2.

  Module ID: E1_OUTCOME_BUCKETIZER
  Symbol: Outcome Bucketizer
  Sub-modules / Key Functions: Classify closed position as outcome 1–6
    (WIN_LARGE, WIN_SMALL, BE, LOSS_SMALL, LOSS_LARGE, TIMEOUT)
  Input → Output: OutcomeRecord → OutcomeBucket
  ────────────────────────────────────────
  Module ID: E2_PROXIMITY_EVALUATOR
  Symbol: Proximity Evaluator
  Sub-modules / Key Functions: Classify next event proximity:
    IMMEDIATE/NEAR/MEDIUM/FAR
  Input → Output: CalendarEvent + ScheduleTick → ProximityBucket
  ────────────────────────────────────────
  Module ID: E3_MATRIX_LOOKUP
  Symbol: Matrix Lookup
  Sub-modules / Key Functions: Query original_combinations /
    reentry_combinations tables; return action, multiplier, delay,
    combination_id
  Input → Output: OutcomeBucket + ProximityBucket + SignalType +
    ChainPosition → MatrixAction
  ────────────────────────────────────────
  Module ID: E4_REENTRY_INTENT_BUILDER
  Symbol: Reentry Intent Builder
  Sub-modules / Key Functions: Build ReentryDecision; enforce
    MaxReentryGenerations=2, MinReentryConfidence; spread guard
  Input → Output: MatrixAction → ReentryDecision → Redis
    reentry_decisions

  Default Matrix Rules (priority order):
  1. Future Proximity → if IMMEDIATE, HOLD/CLOSE
  2. Flash Move → if recent high-volatility, conservative
  3. Regional Equity Open → EQUITY_OPEN signal override
  4. Anticipation Timing → 1HR/8HR window gating
  5. Generation Limit → hard stop after R2
  6. Conservative Fallback → default HOLD
