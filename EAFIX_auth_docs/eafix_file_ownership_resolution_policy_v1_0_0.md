# EAFIX File Ownership Resolution Policy v1.0.0

This document defines the deterministic policy for assigning files within the EAFIX repository to atomic modules.  It supports manifest generation by ensuring that each file is either owned, shared, allowed or forbidden for each module.  Ownership mappings are versioned and must be updated whenever files are added, renamed, removed or reallocated.

## Purpose

Atomic modules are defined by their identity, responsibilities and the code, configuration, documentation and evidence that implement them.  To avoid ambiguity and reduce coupling, every file in the repository must be associated with one or more modules or explicitly designated as shared or forbidden.  This policy describes how to assign files based on their path, function and usage, and how to resolve conflicts when multiple modules could claim the same file.

## File categories

Files are grouped into the following categories, each with its own assignment rules:

| Category | Description |
| --- | --- |
| **Source** | Implementation files: code in languages such as `.py`, `.rs`, `.ts`, `.js`, `.cpp`, `.go` and `.mql`. |
| **Test** | Unit, integration and contract tests, usually located under `tests/` or with suffix `_test` or `_spec`. |
| **Documentation** | Markdown, reStructuredText or plain text files (`README.md`, `CONTRACTS.md`, `STATE.md`, `API.md`, `VERIFICATION.md`, design notes). |
| **Configuration** | Files containing configuration parameters (`.yaml`, `.yml`, `.toml`, `.json`, `.ini`, `.cfg`, `.env`). |
| **Schema** | JSON Schema, Protobuf or Avro definitions (`.json`, `.proto`, `.avsc`) describing data contracts. |
| **Generated** | Files generated by the build process (artifacts, compiled binaries); these should not be committed but may be captured in evidence. |
| **Evidence** | Artifacts used to prove compliance with gates (coverage reports, test results, security scans). |
| **Scripts** | Helper scripts for build, deployment, data migration or tooling (`.sh`, `.ps1`, `.bat`, `.py`). |
| **Third‑party** | Vendored dependencies or external code; ownership is typically `forbidden` to modules. |
| **Migration & crosswalk** | Documents used in architecture evolution (`atomic_module_crosswalk.current.json`, `atonic to canonical crosswalk.csv`); owned by governance. |

## Ownership classes

Each file is classified per module as one of:

| Class | Meaning |
| --- | --- |
| **Owned** | The module has full responsibility for the file.  It may modify the file subject to review and gating.  Only one module can own a file. |
| **Shared** | The file is used by multiple modules and has no single owner.  Modifications require consensus across all using modules and must follow shared kernel governance. |
| **Allowed** | The module may read or import the file but must not modify it.  Allowed files include shared configuration templates, common data definitions and external libraries. |
| **Forbidden** | The module must not reference or modify the file.  Typically applies to files that belong exclusively to another module or to third‑party packages. |

The `assignment_status` in the manifest is derived from the presence or absence of owned files: `owned` if at least one file is owned, `partial` if only shared files exist, `unassigned` if neither owned nor shared.

## Assignment rules

### 1 – Explicit mapping

A file‑to‑module mapping file (`module_file_mapping.csv` or `.json`) SHALL be the primary authority for ownership.  Each record associates a relative path with one or more modules and indicates whether the file is owned, shared or allowed.  When the mapping exists, its assignments override any heuristics.  If a file is not listed in the mapping, apply the rules below.

### 2 – Service home and namespace heuristics

- **Service namespace**: If the file path begins with `services/<service_name>/` or `src/<service_name>/`, and `<service_name>` corresponds to a module’s `deployable_scope`, assign ownership to that module.  For example, `services/configuration_service/src/config/preferences/manager.py` is owned by `F1_CONFIG_PREFERENCES`.
- **Module namespace**: If the file path includes the canonical module symbol or its abbreviation (e.g. `f1`, `d2`), assign ownership accordingly.
- **Testing**: Test files in `tests/<module_symbol>/` or `tests/<service_name>/` are owned by the same module that owns the implementation being tested.
- **Configuration**: Configuration files under `config/<service_name>/` belong to the module’s service.  Global configuration templates (`config/templates/`) are allowed to all modules.
- **Schema & contracts**: Schema files under `contracts/<module_symbol>/` or `schemas/<module_symbol>/` are owned by the module.  Generic schema libraries under `shared/` are shared.
- **Documentation**: Module‑specific documentation (`docs/<module_symbol>/`, `<module_symbol>_README.md`, etc.) is owned.  Global docs (root `README.md`, `docs/governance/`, `docs/architecture/`) are allowed or shared.
- **Scripts**: Deployment and build scripts under a service’s directory (`scripts/<service_name>/`) are owned.  Global scripts are allowed or shared depending on usage.

### 3 – Shared code and libraries

- Files under `shared/` or `lib/` are shared by default.  They belong to the domain group `SK` (shared kernel) and should be mapped to the module `SK1_PLUGIN_INTERFACE` or other shared kernel modules.  Modules consuming shared code mark those files as `shared_files`.
- If a file in `shared/` is used only by a single module, consider moving it to that module’s service or update the mapping to `owned`.

### 4 – UI and gateway files

- Files under `ui/` or `gateway/` are owned by UI modules (`U1_DASHBOARD_BACKEND`, `U2_GUI_GATEWAY`, etc.).  Other modules mark them as `forbidden`.
- Shared components under `ui/shared/` are shared.

### 5 – Broker and plugin files

- Files under `mt4/` or with extension `.mql` belong to broker modules (`B1_MT4_ADAPTER_TRANSPORT`, `B2_MT4_EA_EXECUTOR`) or plugin interfaces (`SK1_PLUGIN_INTERFACE`).  Modules outside the broker domain must not modify these files (`forbidden`).

### 6 – Third‑party and vendored code

- Files under `third_party/`, `vendor/` or similar directories are not owned by any module and must not be modified; they are `forbidden` for all modules.

### 7 – Generated and evidence files

- Generated artifacts (e.g. compiled binaries, auto‑generated code) are not tracked for ownership; they should be ignored in the mapping.
- Evidence files under `evidence/` are associated with the module whose tests or gates produced them; assign them as `owned` or `shared` accordingly.

### 8 – Migration and crosswalk documents

- Files such as `atomic_module_crosswalk.current.json`, `atonic to canonical crosswalk.csv`, `EAFIX_ATOMIC_MANIFEST_POPULATION_FIX_PLAN_V1_0_0.txt` belong to the governance domain and are not owned by modules.  They are `forbidden` for module modification but `allowed` for reading.

### 9 – Conflict resolution

When a file could belong to multiple modules, apply the following:

1. If the file contains code or configuration tied to a specific module’s key function (per the canonical module catalog), assign to that module.
2. If the file is a generic utility used by several modules, mark as `shared`.
3. If the file is entirely within one service’s namespace, assign to that module even if others import it.
4. If no clear ownership exists, mark as `partial` for the candidate modules and flag `needs_review` in their manifests.

### 10 – Logging and provenance

For every file assignment, record:

- The relative file path.
- The module(s) and class (`owned`, `shared`, `allowed`, `forbidden`).
- The evidence supporting the assignment (service namespace, import analysis, explicit mapping).
- The commit hash when the assignment was last updated.

The ownership mapping itself is versioned.  Each time the mapping changes, increment the policy version and update the `module_file_mapping` file accordingly.  Manifest regeneration must be triggered whenever the file mapping changes.

## Examples

### F1_CONFIG_PREFERENCES

Owned files:

- `services/configuration_service/src/config/preferences/` – all implementation code for preferences and configuration.
- `services/configuration_service/tests/config/preferences/` – tests for preferences.
- `services/configuration_service/config/preferences.toml` – configuration file.
- `services/configuration_service/README.md` – module documentation.

Shared files: none.

Allowed files:

- `shared/` libraries used by the configuration service.
- Global docs under `docs/`.

Forbidden files:

- `services/scheduler_service/` – scheduler files (belong to `F3_CLOCK_SCHEDULER`).
- `services/calendar_ingestor/` – calendar modules (`D2`, `D3`).
- `ui/`, `mt4/` – UI and broker modules.

### F3_CLOCK_SCHEDULER

Owned files:

- `services/scheduler_service/src/scheduler/clock/` – scheduling algorithms and clocks.
- `services/scheduler_service/tests/scheduler/` – tests for scheduler.
- `services/scheduler_service/scheduler_config.yaml` – configuration.

Shared files: none.

Allowed files:

- Shared libraries under `shared/`.
- Global docs.

Forbidden files:

- Files owned by configuration, calendar ingest and other modules.

### D2_CALENDAR_SOURCE_ADAPTER

Owned files:

- `services/calendar_ingestor/src/calendar/adapter/`
- `services/calendar_ingestor/src/calendar/ingestor/`
- `services/calendar_ingestor/tests/calendar_adapter/`
- `services/calendar_ingestor/config/ingest_config.yaml`

Shared files:

- `services/calendar_ingestor/shared/calendar_ingestor.py` – shared between `D2` and `D3`.

Allowed files:

- Shared libraries used by the adapter.
- Docs in `docs/`.

Forbidden files:

- `services/configuration_service/`
- `services/scheduler_service/`
- `mt4/`, `ui/`.

Assignment status: `partial` (because of shared files).

### D3_CALENDAR_NORMALIZER

Owned files:

- `services/calendar_ingestor/src/calendar/normalizer/`
- `services/calendar_ingestor/tests/calendar_normalizer/`

Shared files:

- `services/calendar_ingestor/shared/calendar_ingestor.py` – shared.

Allowed files: similar to D2.

Forbidden files: as above.

### F2_EVENT_LOG

Owned files:

- `services/event_log/src/event_log/`
- `services/event_log/tests/event_log/`
- `services/event_log/event_log.toml`
- `services/event_log/README.md`

Shared files:

- `shared/logging_utils.py` – if used by multiple modules.

Allowed files:

- Shared libs; global docs.

Forbidden files:

- Non‑logging modules.

## Governance and future changes

- New files MUST be added to the explicit mapping or placed under the correct service namespace.  Modules must not create files outside their assigned areas without submitting a Module Change Request (MCR).
- When files are renamed, moved or deleted, the mapping MUST be updated.  Automation should detect unmapped files and surface them for review.
- Shared files SHOULD be reviewed regularly to determine whether they should be promoted into the shared kernel (`SK`) or pulled into a specific module.
- Ownership of third‑party code is always `forbidden`; modifications require mirroring changes upstream.

## Enforcement

Automated linting and gating MUST verify that code changes do not modify forbidden files and that shared files are modified only with appropriate approvals.  The manifest generator MUST rely on the explicit mapping combined with these rules to populate the `file_ownership` section of each module manifest.
