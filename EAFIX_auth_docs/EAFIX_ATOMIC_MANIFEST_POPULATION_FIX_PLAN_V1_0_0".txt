The previous plan should be converted into an execution spec, not a prose plan. The AI application needs deterministic task IDs, dependencies, target files, patch rules, validation gates, and stop conditions.
The governing facts are:
	•	The atomic manifest schema requires one module packet with required sections like module_identity, process_binding, contracts, file_ownership, dependencies, gates, failure behavior, docs, and staleness policy. It also forbids old permanent work_cells, submodules, work_cell_context, and child_modules fields.  
	•	The project already treats each module as an atomic context unit with co-located artifacts, a single manifest, and numeric identity direction.  
	•	The module reference already identifies the key drift issues: symbolic vs numeric IDs, differing module counts, incomplete file mapping, and the need to use the assignment policy instead of guessing file owners.  
	•		•	The schema supports owned_files, shared_files, unassigned_candidate_files, file_assignment_status, and ownership_derivation, so shared-entrypoint resolution can be represented without corrupting atomic ownership.
